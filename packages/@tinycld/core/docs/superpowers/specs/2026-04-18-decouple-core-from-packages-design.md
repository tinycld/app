# Decouple Core From Mail and Drive Internals

> **⚠️ Historical / superseded.** Frozen design doc from April 2026. The narrative below references Bun as the project package manager — the project has since migrated to pnpm. Kept for historical context.

## Goal

Make a fresh `@tinycld/core` clone with `tinycld.packages.ts = []` typecheck and run cleanly. Today two files in core pull in package internals, preventing the "lean shell" promise of the extraction work:

- `app/a/[orgSlug]/settings/personal.tsx` imports `useDefaultMailbox` from `@tinycld/mail` (needed by `GoogleTakeoutImportSection`)
- `app/share/[token].tsx` imports `PublicSharePage` from `@tinycld/drive`

After this work, both features continue to exist — they just live in the packages that own them, not in core.

## Non-goals

- Removing the features. Both the takeout import and public share links stay functional.
- Changing the takeout import's internals. The parser/worker/batch-inserter machinery under `lib/takeout-import/` is already package-agnostic; it moves verbatim into its new home.
- Generalizing beyond the two specific couplings. If another coupling surfaces later, we address it separately.
- A new plugin discovery system. The existing `packageRegistry` (generated from `tinycld.packages.ts`) is sufficient to know which packages are linked.

## Background

The package-extraction work (`docs/superpowers/specs/2026-04-18-extract-packages-to-sibling-repos-design.md`, executed through `dc18b1b`) moved contacts/mail/calendar/drive out of core into sibling repos. All four can now be linked in or left out independently — *except* that the two files above make mail and drive load-bearing for core's typecheck.

Both couplings are genuine cross-package features, not leaks:

- **Google Takeout Import** imports Google's `.zip` export into contacts/calendar/drive/mail. It inherently depends on whichever of those packages is present. Today that coordination lives in `personal.tsx` inside a 150-line `GoogleTakeoutImportSection` component, plus the `lib/takeout-import/` subsystem.
- **Public Share Links** are how drive lets someone share a file via URL. Today the `/share/<token>` route lives in core but the page component lives in drive.

The existing generator (`scripts/generate-packages.ts`) already wires org-scoped routes (`screens/` → `app/a/[orgSlug]/<slug>/…`). It doesn't yet wire public top-level routes; we add that.

## Design

```
    manifest.ts                              # declares publicRoutes contribution
```

### Part 1 — Extend manifest with `publicRoutes`

The generator currently wires org-scoped routes via `manifest.routes.directory`. Add a parallel `publicRoutes.directory` field. Files under that directory map one-to-one into core's `app/` at the same subpath.

Manifest addition:
```ts
publicRoutes?: { directory: string }
```

Generator behavior (new function paralleling `generateRoutes`):
```ts
function generatePublicRoutes(
    packageName: string,
    manifest: PackageManifest,
    packageDir: string
): string[] {
    if (!manifest.publicRoutes?.directory) return []
    const sourceDir = path.join(packageDir, manifest.publicRoutes.directory)
    if (!fs.existsSync(sourceDir)) return []

    const generated: string[] = []
    for (const relPath of walkScreens(sourceDir)) {
        const withoutExt = relPath.replace(/\.tsx?$/, '')
        const outFile = path.join(ROOT, 'app', `${relPath}`)
        const importPath = `${packageName}/${manifest.publicRoutes.directory}/${withoutExt}`

        fs.mkdirSync(path.dirname(outFile), { recursive: true })
        fs.writeFileSync(
            outFile,
            `export { default } from '${importPath}'\n`
        )
        generated.push(outFile)
    }
    return generated
}
```

The existing `cleanPrevious` pass already removes any file tracked in `.package-links.json` on the next run, so removing a package also removes its public routes.

**Conflict policy:** if two packages declare the same public route path, the generator errors out with both package names listed. No silent wins.

**Opaque file list (important):** the generator must **not** scan or filter by filename. Any file under `public-screens/` is a candidate — this keeps drive's directory opaque to the generator (drive owns the route filenames and subdirectories, not core).

### Part 2 — Move `/share/<token>` into drive

- Move `app/share/[token].tsx` from core to `@tinycld/drive/public-screens/share/[token].tsx` — verbatim, except the import rewrites back to local paths (`~/lib/auth` becomes `@tinycld/core/lib/auth` or stays with the `~` alias if the core's tsconfig path resolution works from the linked sibling; verify during implementation).
- Update drive's `manifest.ts` to add `publicRoutes: { directory: 'public-screens' }`.
- Update drive's `package.json` `exports` map: `"./public-screens/*": "./public-screens/*.tsx"`.
- Delete `app/share/[token].tsx` from core.
- After `pnpm run packages:generate`, core gets an auto-generated `app/share/[token].tsx` containing only `export { default } from '@tinycld/drive/public-screens/share/[token]'`.
- Add `/app/share/` to core's `.gitignore` section for generated wiring — it joins `lib/generated/` and the org-scoped package route re-exports.

**Cross-package import from a sibling back into core:** `PublicSharePage` in drive uses `useAuth` and `PB_SERVER_ADDR` from core. Those imports in the moved file continue to work because drive is `pnpm link`ed under core's `node_modules/` — resolution of `~/lib/auth` happens at core's tsconfig level when drive is compiled through core's build. Keep the imports as-is; don't try to invent a cross-package alias.

### Part 3 — Create `@tinycld/google-takeout-import` package

New sibling repo at `~/code/tinycld/google-takeout-import/`. Structure:

```
@tinycld/google-takeout-import/
    package.json
    manifest.ts
    components/
        GoogleTakeoutImportSection.tsx      # the 150-line component from personal.tsx
        TakeoutIdleState.tsx                # any sub-components that currently live inline
        TakeoutDetectedState.tsx
        TakeoutImportingState.tsx
        TakeoutFinishedState.tsx
    hooks/
        useDefaultMailbox.ts                # moved from @tinycld/mail
    lib/
        (the 11 files from core's lib/takeout-import/ — batch-inserter, parsers/, worker*, etc.)
    types.ts                                # has ImportService type
    settings/
        takeout.tsx                         # the settings panel entry point
```

**Manifest** uses the existing `settings` contribution field (see `packages/mail/manifest.ts` for a reference — mail also contributes a settings panel):

```ts
const manifest = {
    name: 'Google Takeout Import',
    slug: 'google-takeout-import',
    version: '0.1.0',
    description: 'Import Google data (Mail, Contacts, Calendar, Drive) from Google Takeout .zip files.',
    nav: null,                              // no top-level nav — appears only in settings
    settings: [
        {
            slug: 'google-takeout',
            component: 'settings/takeout',  // resolves to @tinycld/google-takeout-import/settings/takeout
            label: 'Import from Google',
        },
    ],
}
```

The generator already handles the `settings` field — this package piggybacks on existing infrastructure.

**`nav: null`** — the package has no dedicated top-level nav entry or screen list. It's pure a settings-panel contribution. Verify the existing generator handles `nav: null` cleanly; if not, make it handle it cleanly. (This is a small extension; confirm during implementation.)

**`useDefaultMailbox` is duplicated, not moved.** Mail uses the hook internally (`InlineReply.tsx`) — it can't leave mail. The hook is 12 lines and queries the `mail_mailbox_members` collection. The takeout package reimplements it against a *locally declared* minimal row type so the takeout package typechecks whether or not mail is linked:

```ts
// @tinycld/google-takeout-import/hooks/useDefaultMailbox.ts
import { PocketBase } from 'pocketbase'
import { useAuth } from '~/lib/auth'  // via core

type MailMailboxMember = { user_org: string; mailbox: string }

export function useDefaultMailbox(): string | null {
    const { userOrgId, pb } = useAuth()
    const [members, setMembers] = useState<MailMailboxMember[]>([])
    useEffect(() => {
        if (!userOrgId) return
        pb.collection('mail_mailbox_members')
            .getList<MailMailboxMember>(1, 1, { filter: `user_org="${userOrgId}"` })
            .then(r => setMembers(r.items))
            .catch(() => setMembers([]))
    }, [userOrgId, pb])
    return members[0]?.mailbox ?? null
}
```

When mail isn't linked, PocketBase returns an error (collection doesn't exist) which `.catch()` swallows, yielding `null`. Both copies read the same underlying collection, which is mail's published schema contract. If mail's schema changes, both copies need updating — rare enough to accept as ordinary duplication cost.

Verify during implementation: the exact `useStore`/`useOrgLiveQuery` path inside mail's copy continues to work (it will, since when that code runs, mail is linked and its collection is in the schema). The takeout copy uses the looser typed-PocketBase-client approach above.

### Part 4 — Runtime package discovery for the mail checkbox

The takeout section previously showed a "Mail" checkbox only when `useDefaultMailbox()` returned a non-null value. Semantically that was "is mail linked + configured?"

In the new structure, the component runs inside `@tinycld/google-takeout-import`, and it should check whether `@tinycld/mail` (and each other importable package) is actually linked. Use the existing `usePackages()` hook from `~/lib/packages/use-packages`:

```ts
import { usePackages } from '@tinycld/core/lib/packages/use-packages'  // or equivalent

function GoogleTakeoutImportSection() {
    const packages = usePackages()
    const installedSlugs = new Set(packages.map(p => p.slug))

    const mailAvailable = installedSlugs.has('mail')
    const mailboxId = mailAvailable ? useDefaultMailbox() : undefined
    const detectedServices: ImportService[] = []
    if (detection?.hasContacts && installedSlugs.has('contacts')) detectedServices.push('contacts')
    if (detection?.hasCalendar && installedSlugs.has('calendar')) detectedServices.push('calendar')
    if (detection?.hasDrive && installedSlugs.has('drive')) detectedServices.push('drive')
    if (detection?.hasMail && mailAvailable && mailboxId) detectedServices.push('mail')
    // …
}
```

**Rules of hooks:** call `useDefaultMailbox()` unconditionally. Its internals query `user_org.mailbox` — when mail isn't linked, mail's migration hasn't run, so that column doesn't exist and the query returns undefined. Reading a missing column through pbtsdb's schema-typed query is safe (returns undefined); reading it through a raw PocketBase query could error, so keep the implementation on `useOrgLiveQuery`. Verify this during implementation; if the query actually errors, wrap the read in a try/undefined rather than adding a conditional hook call.

### Part 5 — Core becomes lean

After the moves:

- `lib/takeout-import/` is gone from core (moved into the takeout package)
- `GoogleTakeoutImportSection` is gone from `personal.tsx` (moved into the takeout package; settings panel generator wires it back in)
- `app/share/[token].tsx` is gone (generated by the packages generator from drive's `publicRoutes`)
- `app/a/[orgSlug]/settings/personal.tsx` has no package imports — the takeout section is wired through the existing `package-settings.ts` mechanism that each package's manifest `settings` field contributes to

Verify: `grep -r '@tinycld/' app/ lib/` in core returns zero results (modulo generated files under `lib/generated/`).

### Data flow

**Takeout import:**
1. User opens Personal Settings, sees "Import from Google" section because `@tinycld/google-takeout-import` is linked and contributes a settings entry.
2. Section calls `usePackages()` and `useDefaultMailbox()` — returns `undefined` when mail isn't linked.
3. User picks a `.zip`; `useTakeoutImport()` drives parse/worker/batch-insert.
4. Services filtered by `installedSlugs.has(slug)` — only installed packages appear as options.

**Public share link:**
1. User visits `/share/<token>`.
2. Core's `app/share/[token].tsx` (generated) re-exports `@tinycld/drive/public-screens/share/[token]`.
3. Page does auth check, resolves token to `{org_slug, item_id}`, redirects logged-in users or renders `PublicSharePage`.

### Error handling

- **Takeout package missing:** the settings panel entry isn't contributed; Personal Settings has no "Import from Google" section. Clean graceful degradation.
- **Drive missing but share URL visited:** the generated `app/share/[token].tsx` won't exist (the generator only creates it when drive is linked). Expo Router returns its default 404. No runtime error.
- **Mail missing but user tries to import mail data:** the checkbox won't render because `installedSlugs.has('mail')` is false. The takeout detection still detects mail data in the zip, and we surface a "Mail data detected but @tinycld/mail is not installed — install it to import" banner. (Optional polish; document but don't block on it.)
- **Public route collision between two packages:** generator throws with both package names.

### Testing

- **Unit:** `usePackages()` filtering test — given a registry with contacts+calendar linked, `installedSlugs.has('mail')` is false. (Test in isolation with a mocked registry.)
- **Public route generation:** add a case to the generator test suite where a fixture package declares `publicRoutes.directory` and the generator emits an `app/share/[token].tsx` re-export.
- **Integration smoke:** with takeout package + mail linked, the Personal Settings "Import from Google" section appears; without takeout linked, it doesn't. Verified manually per the Task 5 pattern used in the extraction work.
- **Typecheck from empty state:** with `packages = []`, `pnpm run typecheck` exits 0. This is the load-bearing assertion.

## Alternatives considered

- **Guard imports with `typeof import('@tinycld/mail')`:** TypeScript doesn't fully support conditional module resolution; even if it worked at the type level, the runtime side requires lazy `import()` which ripples through the component tree. Too much complexity for the payoff.
- **Leave `/share/<token>` in core, 404 when drive not linked:** cheap but doesn't fix the typecheck. Core still has a hard-type dep on `@tinycld/drive`.
- **Merge the takeout import into an existing package (e.g. mail):** it's *not* just a mail thing — it imports contacts/calendar/drive data too. Making any one of those four packages the owner is wrong; a dedicated coordinating package is the right shape.

## Migration plan summary (for plan step)

1. Add `publicRoutes` support to the generator (new code + test).
2. Move `/share/<token>` into drive (new `public-screens/`, manifest update, exports map update, delete from core).
3. Create `@tinycld/google-takeout-import` sibling repo, populate from core's existing files.
4. Replace `GoogleTakeoutImportSection` in `personal.tsx` with nothing — let the settings panel mechanism pull it in.
5. Delete `lib/takeout-import/` from core.
6. Delete the `useDefaultMailbox` usage and import from core (or more precisely, it's already gone once the component moves).
7. Verify `packages = []` typechecks.
